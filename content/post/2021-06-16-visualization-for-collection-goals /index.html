In September 2019, Mia wrote ‚ÄúMia Collection Development Plan 2020‚Äù for 2020 and 2021. In this document, each department reviewed their current collection and made some goals for new collections for the next 2 years, especially focusing on some diverse classifications they would like to have. Now, it‚Äôs June 2021, Mia starts to think about some ways they can sufficiently evaluate how their collection development goals have been achieved.

To quantitatively evaluate their progress, Prof.¬†Brianna and I met with Mia‚Äôs data team and talked about making visualizations for each department at Mia to check the growth or trend in their collections. We made pretty simple ones in the beginning to check if visualizations work better for them compared with traditional tabled data.

For example, as the graph shown below, to evaluate if department A has increased their certain material types‚Äô or object types‚Äô collections recently, we could generate one visualization like this:

department.A.2015.2020 %>% 
  filter(Classification.X == TRUE | Classification.Y == TRUE) %>%
  ggplot(mapping = aes(x = AccessionISODate, fill = continent)) + 
  geom_histogram(binwidth = 50) +
  theme_classic()

What if we would like to check data for another two classification types? Then we just simply change the filters in the code. Here we go again:

department.A.2015.2020 %>% 
  filter(Classification.Z == TRUE | Classification.W == TRUE) %>%
  ggplot(mapping = aes(x = AccessionISODate, fill = continent)) + 
  geom_histogram(binwidth = 50) +
  theme_classic()

The only difference between these two chunk of codes is this line:

filter(Classification.X == TRUE | Classification.Y == TRUE) 
filter(Classification.Z == TRUE | Classification.W == TRUE)

All other codes are identical and we only need to change the classification type each time. After a couple trying, we realized that we are able to code a general function in R to easily change some filters without unnecessary repetitions, and the demo for the function look like this:

demoFunction <- function(StartDate,EndDate,Dept,Class){
 
  fullData %>% 
    filter(department == Dept) %>%
    filter(AccessionISODate >= as.Date(StartDate), AccessionISODate <= as.Date(EndDate)) %>%
    dplyr::filter(eval(parse(text=paste0(paste0('Classification.',str_replace_all(Class,' ','_')),collapse='|')) )) %>% 
    ggplot(mapping = aes(x = AccessionISODate, fill = continent)) + 
    geom_histogram(binwidth = as.numeric((as.Date(EndDate)-as.Date(StartDate))/30)) +  
    scale_color_viridis_d() +
    theme_classic()
}

demoFunction("2015-01-01","2020-01-01","A", c("X", "Y", "Z", "W"))

It is still a function in process. But you got the idea. We are working on leaving more freedom to users to filter and choose the aspects they would like to focus on (color the graph). Hence, a Shiny App would be the best try.

Wait a minute‚Ä¶ Why? What is the pros and cons of allowing users to filter and color the graph?

First, some departments have very specific collection goals, for example, one department would like to check large-scale painting from Country T. To evaluate this goal, we need to include very specific ‚ÄúDimensions + Classification + Country of origin‚Äù accession data for them. But there are a large number of specific goals. So making a shinyApp would allow each department to filter the date, classification, country, dimension, style‚Ä¶ directly depending on their needs. And they could do it over and over again to narrow down the filtering range.

But most importantly, establishing the Shiny App and having recursive functions might be the starting point of making a sustainable way to evaluate Mia‚Äôs collection goals. Instead of rewriting the similar code over and over every 2 years or going through the metadata all the time, Mia only needs to simply update the database in the future. In the best scenario, establishing a sustainable and efficient web for data visualizations will help Mia in the long term.

Now, I have made some progress in thinking about the need of clients:

  • write functions when you see repetitive codes
  • create user friendly visualization for non-data clients
  • start to think about sustainable visualization might be re-used in the future
